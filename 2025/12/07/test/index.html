Writeup (网络运维小助手-rag)

  • Flag1（jailbeak day by day）：200 分
  • Flag2（retrieval can’t save generation）：200 分

题目描述

小光是一位新入职的网络安全管理员，负责一些基本的运维相关工作。为了追逐大模型应用的热潮，他也开发了基于检索增强生成（RAG）与大语言模型的网络运维小助手。为了分担自己的工作压力，小光将两个密钥以System Prompt的方式交给助手进行保管。

小光并非提示词工程高手，但他对大语言模型的提示词越狱有所耳闻。为了防止密钥被窃取，他设计了若干防护措施，阻止他能想到的提示词攻击方式。看起来应该万无一失了……

果真如此吗？

挑战目标

你需要通过与大语言模型的交互获取 Flag。Flag 即为大语言模型保管的“密钥”。

  • Flag 1 : 在不使用知识库的情况下，通过构造输入，从助手的输出获取 Flag。
  • Flag 2 : 在使用知识库的情况下，通过构造输入，从助手的输出获取 Flag。

特别提示

  • 输入的长度上限为 300 字。
  • 不支持流式输出，但限制了输出 token 上限。请耐心等待输出结果。
  • 每位选手每 10 秒只能尝试与模型交互 1 次，比赛期间每位选手总共只能尝试 500 次。

若干防护措施？

  • Flag1：
    • 在 system prompt 中简单叮嘱大语言模型不可泄密 flag。
    • 基础静态关键词审核，输出中包含flag则会拒绝输出。（但没有屏蔽大写的 flag）
  • Flag2：
    • Flag1 的全部防护措施
    • 对 RAG 向量相似度召回设定阈值，如无任何知识达到与问题的相似度阈值，自动拒答以规避无关 prompt。

Flag2

解决 Flag1 后，根据题目提示，需要打开知识库开关进行带 RAG 的问答。再试试看第一问通过的 prompt，系统提示“抱歉，无相关参考资料，请您换一个问题吧”。

根据回复内容，猜测系统有无召回拒答策略。因此需要先探测一下知识库到底有什么内容。

大模型笑传之查查弊

作为静态 flag+易留存交互记录的题目，大模型题可谓是检查违反诚信比赛须知的有力武器。

被判定为作弊的情况包括但不限于：

  • 提交了 flag，但根本没有和模型（对应 flag 的模型）对话过
  • 提交了 flag，但 prompt 与他人雷同
  • 你可能想不到的其他方式？

其它有关/想说的

  • 本来计划启用防护更强的提示词/大模型，但确实攻破难度提升太多，有点偏离题目娱乐题的难度定位，于是换了现在的版本。（于是成功拿下分数衰减最高题目）
  • 大语言模型输出确有随机性，拼尽全力难以避免同 prompt 不同输出的情况。
  • RAG 的拒答相似度阈值似乎有些玄学，再加上确实都是和网络安全相关的知识，导致可能没有一视同仁地卡住所有人第一问的 Prompt。可能造成了难度上的不平衡，下次得考虑放个和计算机科学/网络安全没那么相关的知识库了 Orz
  • 系统的实际使用量完全没有达到赛前根据参赛人数预期的值。早知道不花时间改全异步了 QAQ
